**Project:** Controle de Vendas Premium V2

## Current State

The project is a Brazilian sales control application with:
- Firebase Auth (anonymous + custom password-based login)
- Firestore for sales, clients, contracts, and routine state
- Firebase Storage for contract PDFs
- 4 user roles (ADM, Manager, Gabriela, Sabrina) with hash-based auth
- PWA with offline support
- ~60+ components, 38 hooks, 6 services

---

## CRITICAL Findings (Immediate Action Required)

### 1. `.env` File with Real Credentials on Disk
- **Location:** `.env` at project root
- **Risk:** Firebase API keys + SHA-256 password hashes exposed locally
- **Mitigation:** Verify `.env` never committed to git history. If committed, rotate all credentials immediately.

### 2. Legacy SHA-256 Password Hashes
- **Location:** `.env` (VITE_ADM_HASH, VITE_HASH_GABRIELA, VITE_HASH_SABRINA, VITE_MANAGER_HASH)
- **Risk:** Raw SHA-256 with no salt â€” vulnerable to rainbow table attacks
- **Mitigation:** Regenerate all hashes using `hashPassword()` from `src/utils.js` (PBKDF2, 310k iterations, 16-byte salt)

### 3. Predictable ID Generation
- **Location:** `src/utils.js:generateLocalId()`
- **Risk:** Uses `Date.now() * 1000 + Math.random()` â€” not cryptographically secure
- **Mitigation:** Replace with `generateUUID()` (uses `crypto.randomUUID()`)

---

## HIGH Findings (Should Be Addressed Soon)

### 4. No Per-User Data Isolation in Firestore Rules
- **Location:** `firestore.rules`
- **Risk:** Any authenticated user can read ALL sales and ALL client data (including CPFs, phones, addresses)
- **Mitigation:** Add per-user filtering rules based on `employeeName` or user role

### 5. No Ownership Validation on Writes
- **Location:** `firestore.rules`
- **Risk:** Any authenticated user can modify any sale or client record
- **Mitigation:** Validate `employeeName` matches authenticated session or user has admin/manager role

### 6. Hardcoded Admin Name Fallback
- **Location:** `src/constants.js`
- **Code:** `import.meta.env.VITE_ADM_NAME || 'Dark Morellato'`
- **Risk:** Leaks admin name if env var is missing
- **Mitigation:** Remove fallback or use generic placeholder

### 7. Missing Firestore Rules for `system_logs`
- **Location:** `firestore.rules`
- **Risk:** `authService.logAction()` writes fail silently (default deny rule blocks writes)
- **Mitigation:** Add explicit rules allowing authenticated users to create logs

---

## MEDIUM Findings

### 8. Potential Firestore Rules Syntax Error
- **Location:** `firestore.rules` â€” `rotina_state` write rule
- **Code:** `request.time.toDate().format('YYYY-MM-DD')`
- **Risk:** `format()` may not be valid Firestore Rules syntax
- **Mitigation:** Test rule or use alternative date comparison

### 9. Backup Import Has Minimal Validation
- **Location:** `src/services/backupService.js`
- **Risk:** Only validates `id` is string â€” malformed data could enter Firestore
- **Mitigation:** Add schema validation for sale/client structure

### 10. Contracts Stored as Base64 in Firestore
- **Location:** `src/services/salesService.js`
- **Risk:** Increases Firestore read costs, less efficient than Storage
- **Mitigation:** Migrate to Firebase Storage with signed URLs

### 11. Empty `schemas/` Directory
- **Location:** `src/schemas/`
- **Risk:** No validation schemas for sale/client data before Firestore writes
- **Mitigation:** Add Zod or Yup validation schemas

---

## LOW Findings

### 12. `checkJs: false` in tsconfig
- **Risk:** JavaScript files not type-checked
- **Mitigation:** Enable `checkJs: true` gradually with JSDoc annotations

### 13. No Security-Specific ESLint Rules
- **Risk:** Missing `no-eval`, `no-implied-eval`, `no-new-func` rules
- **Mitigation:** Add security lint rules to `eslint.config.js`

### 14. Minimal Test Coverage
- **Risk:** Only 5 test files exist â€” most code untested
- **Mitigation:** Add tests for auth, services, and security-critical utilities

## Implementation Plan

### Phase 1: Critical Fixes (Day 1)
1. Regenerate all password hashes to PBKDF2 format
2. Replace `generateLocalId()` with `generateUUID()`
3. Verify `.env` is not in git history
4. Remove hardcoded admin name fallback

### Phase 2: High Priority (Day 2-3)
5. Fix Firestore rules for per-user data isolation
6. Add ownership validation on writes
7. Add `system_logs` collection rules
8. Add role-based access control in Firestore rules

### Phase 3: Medium Priority (Day 4-5)
9. Fix `rotina_state` rule syntax
10. Add backup import validation schemas
11. Create validation schemas in `src/schemas/`
12. Add security ESLint rules

### Phase 4: Low Priority (Day 6+)
13. Enable `checkJs: true` in tsconfig
14. Expand test coverage for security-critical code
15. Consider migrating contracts to Firebase Storage

---

## Tradeoffs

| Decision | Pro | Con |
|---|---|---|
| Per-user Firestore rules | Better data isolation | More complex rules, potential performance impact |
| PBKDF2 hash migration | Stronger security | Requires all users to re-set passwords or dual-verify during transition |
| Contract storage migration | Lower Firestore costs | Requires data migration, Storage rules update |
| Full TypeScript migration | Compile-time safety | Significant refactoring effort |
